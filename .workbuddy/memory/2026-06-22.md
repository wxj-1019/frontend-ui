## Next.js 构建类型错误修复

### 根因分析

所有组件的 `className` 被推断为 `required` 的根因：

1. **tsconfig paths 映射**: `@frontend-ui/ui` → `../packages/ui/src` 让 TypeScript 直接读取源文件
2. **`@/lib/utils` 无法解析**: `packages/ui/src` 中的源文件引用 `@/lib/utils`，但 docs 的 `@/*` 映射到 `./*`（即 apps/docs/），导致 `clsx`/`tailwind-merge` 在 apps/docs 上下文中找不到
3. **类型坍塌**: 当 `@/lib/utils` 解析失败时，整个组件的 props 类型坍塌为 `{ children: any; className: any; ... }`，`className` 被标记为 required

### 最终修复方案

**取消 tsconfig paths 映射到源文件，改用 dist 的 .d.ts 文件**：

1. 将 `@frontend-ui/ui` 的 tsconfig paths 改为指向 `../packages/ui/dist`
2. 使用 `tsc --declaration --emitDeclarationOnly` 生成 DTS 文件（而非 tsup 的 DTS worker，后者会 OOM）
3. **关键**: tsup 的代码分割会为每个 entry 生成 `.js`/`.mjs` 文件（如 `dist/animations/fade-content.js`），这些文件与 `dist/animations/fade-content/` DTS 子目录同名，TypeScript bundler resolution 优先匹配 `.js` 文件，忽略子目录中的 `index.d.ts`
4. **DTS 重定向**: 为每个被遮蔽的组件创建同层 `.d.ts` 文件，内容为 `export * from "./{name}/index"`，强制目录解析优先于文件解析
5. **补充**: 删除 `apps/docs/lib/utils.ts`（已不再需要，但保留了以防万一）

### 构建工作流更新

packages/ui 构建需要两步：
```bash
cd packages/ui
rm -rf dist
SKIP_DTS=1 npx tsup                    # 生成 JS 产物（跳过 DTS 避免 OOM）
npx tsc --declaration --emitDeclarationOnly --outDir dist  # 生成 DTS 声明
python scripts/generate-dts-redirects.py                     # 生成 DTS 重定向文件
```

### 禁用 standalone 模式

Windows 上 Next.js 15 的 `output: 'standalone'` 在 copy traced files 阶段因 symlink 权限问题（EPERM）失败，已在 next.config.ts 中移除该配置。
